# Loop Engineering 使用文档

> v0.1.15 | 2026-07-03

---

## 1. 前置条件

在使用 Loop Engineering 之前，请确认以下条件已满足：

| 条件 | 说明 |
|------|------|
| **Windows 系统** | 桌面端目前仅支持 Windows |
| **Git 仓库** | 你的项目必须有 `.git` 目录（`git init` 或 `git clone`） |
| **Claude Code** | 需自行安装，Loop 通过 `claude` 命令行驱动 agent |

> **注意**：`tasks.md` 和 `loop-config.yaml` 不需要手动创建，系统会在添加任务和运行 Setup 时自动生成。

## 2. 安装 & 首次启动

### 2.1 安装

从 GitHub Releases 页面下载最新 `Setup.exe`，双击安装。

安装完成后，桌面端以**系统托盘**方式运行，没有主窗口。

### 2.2 首次启动

首次启动时，设置面板会自动弹出：

```
┌─────────────────────────────────┐
│     Loop Dashboard 设置          │
│                         v0.1.15  │
│ ─────────────────────────────── │
│ 端口号: [8765]                   │
│ 修改端口后需重启生效              │
│                                 │
│ ☑ 开机自启                      │
│ ☑ 启动时自动打开 Dashboard       │
│ ─────────────────────────────── │
│           [保存]  [取消]         │
└─────────────────────────────────┘
```

| 选项 | 说明 |
|------|------|
| **端口号** | Dashboard 服务端口，默认 8765。被占用时自动尝试下一个端口（最多 5 次） |
| **开机自启** | 勾选后随 Windows 启动 |
| **自动打开浏览器** | 勾选后每次启动自动打开 Dashboard |

> 后续可以通过右键托盘 → **设置** 重新打开此面板。

## 4. 添加项目

### 4.1 打开 Setup 页面

右键托盘 → **添加项目**，浏览器打开 Dashboard 的 `/setup` 页面：

1. **选择项目类型**：Generic（通用）、Unity+ToLua（Unity 游戏项目）、Python Server
2. **填写项目根目录**：你的 git 仓库路径（如 `d:/work_pvp/PVPProject8`）
3. **Agent 工作区目录**：Agent 的 worktree 存放位置，选项目根目录后自动填入
4. **Agent 名称**：留空则自动读取 `git config user.name`
5. （Unity 项目）**MCP 端口**：主工程默认 8080，Agent 默认 9080

### 4.2 点击"运行 Setup"

Setup 自动完成以下步骤：

1. 创建 Agent git worktree（独立工作目录，agent 并行不冲突）
2. 生成 MCP 配置文件（`.mcp.json`、`McpProjectConfig.json`）
3. 部署 Claude Code skills 和 commands（`.claude/skills/`、`.claude/commands/`）
4. 部署环境脚本（`.claude/scripts/`）
5. 生成人工验证清单（`TEST.md`）
6. 同步配置到 Agent worktree

Setup 完成后自动跳转到项目 Dashboard。

## 5. 任务管理

### 5.1 任务生命周期

```
                      ┌──────────┐
                      │   [ ]    │  待办（pending）
                      │   待办   │  等待 agent 抢取
                      └────┬─────┘
                           │ task-runner 抢到任务
                           │ 标记 [~] 进行中
                           ▼
                      ┌──────────┐
              ┌──────►│   [~]    │  进行中（in_progress）
              │       │  进行中  │  implementer 实现 + verifier 验证
              │       └────┬─────┘
              │            │
              │            ├── implementer + verifier PASS
              │            │   taskhelper run-done 标记 [x]
              │            ▼
              │       ┌──────────┐
              │       │   [x]    │  完成（done）
              │       │   完成   │  等待合入或已合入
              │       └────┬─────┘
              │            │
              │            ├── 用户不满意，reopen + 反馈
              │            ▼
              │       ┌──────────┐
              │       │   [r]    │  返工（reopen）
              │       │   返工   │  带反馈，agent 下一轮重新抢
              │       └────┬─────┘
              │            │ task-runner 抢到 [r] 任务
              │            │ 在已有分支上继续修改
              │            │
              └────────────┘

        用户手动操作：
        
        [~] 重置 → [ ]     任务被中断（终端被关、进程崩溃），
                          手动重置回待办，下次重新抢
        
        任意状态 → 删除     手动删除任务及关联的 agent 分支
```

### 5.2 添加任务

在 Dashboard 任务页（`/tasks`）底部的表单填写：

- **描述**：具体可执行的动作 + 验收条件
- **指派给**：Agent 名称（如 `with`），只有名字匹配的 agent 才会抢这个任务

提交后 `tasks.md` 自动创建（如果尚不存在），并生成一个 8 位十六进制 task_id。

### 5.3 筛选和查看

任务页支持：

| 筛选维度 | 选项 |
|----------|------|
| **状态** | 待办、进行中、已完成、返工（可多选） |
| **Agent** | 下拉列表，默认选中自己 |
| **排序** | 最新在前 / 最旧在前 |

### 5.4 重置任务——恢复被中断的任务

当任务卡在 `[~]`（进行中）但实际已停止（终端被关、进程崩溃等），点击 **重置** 将其恢复为 `[ ]`（待办）。下一轮 loop 会重新抢取。

### 5.5 返工——让 agent 重新实现

已完成的任务（`[x]`）如果结果不满意，可以 **reopen**：

1. 填写反馈（如"战斗面板的伤害数字没有格式化"）
2. 任务状态变为 `[r]`（返工），反馈写入 tasks.md
3. 下一轮 loop 会抢取，agent 在已有分支上根据反馈继续修改
4. verifier 会根据反馈重点验证之前失败的点

> 如果 3 个 implementer 都不收敛（FAIL 数不降），任务标记 `IMPx(未收敛)`，需要人工介入。

### 5.6 删除任务

删除任务时会同时清理关联的 `agent/*` 分支。

## 6. 运行 Loop

### 6.1 启动

在 Dashboard 控制页（`/control`）点击 **启动 Loop**：

1. 系统打开一个终端窗口，运行 `claude --dangerously-skip-permissions`
2. 自动输入 `/runloop` 命令
3. Loop 开始每 2 分钟轮询一次 tasks.md，抢任务执行

### 6.2 控制操作

| 操作 | 按钮 | 效果 |
|------|------|------|
| **启动** | 🟢 启动 Loop | 打开终端窗口，开始轮询 |
| **暂停** | 🟡 暂停 | 当前任务继续执行，下一轮不抢新任务 |
| **恢复** | 🔵 恢复 | 取消暂停，恢复轮询 |
| **停止** | 🔴 停止 Loop | 关闭终端窗口，终止进程 |
| **聚焦** | 🔍 聚焦窗口 | 找回被其他窗口盖住的 Loop 终端 |

### 6.3 状态指示灯

控制页顶部有一个状态圆点：

| 指示灯 | 状态 | 含义 |
|--------|------|------|
| 🟢 绿色闪烁 | Loop 运行中 | 心跳正常，agent 在工作 |
| 🟡 黄色 | 已暂停 | 暂停中，不抢新任务 |
| ⚫ 灰色 | 空闲 | Loop 未运行 |

心跳时间显示为相对时间（如 "3s ago"、"2m ago"），每 5 秒自动刷新。

### 6.4 聚焦窗口

Loop 的终端窗口可能被其他窗口盖住。点击 **聚焦窗口** 按钮，系统会将终端窗口弹到最前面。

### 6.5 终端意外关闭

如果终端窗口被手动关闭（点了 X）：

- Dashboard 在 5 秒内自动检测到心跳消失
- 状态变为"空闲"
- 点击 **启动** 即可重新开始，之前的任务不会丢失

### 6.6 所有任务完成后

当 tasks.md 中所有任务都标记 `[x]` 后：

- Loop **不会自动停止**，每 2 分钟轮询一次（"起来→发现没任务→退出"）
- 建议：手动停止 Loop 或添加新任务

### 6.7 Loop 卡住排查

如果 Loop 在运行但很久没有产出，从三个维度排查：

| 维度 | 在哪看 | 怎么看 |
|------|--------|--------|
| **心跳** | 控制页顶部 | 心跳是否还在更新？如果显示 "N 分钟前" 且持续增长，Loop 可能挂了 |
| **日志** | 控制页→Loop 日志 | Claude 最近说了什么？是否卡在某一步反复输出？ |
| **任务** | 任务页 | 当前任务是否一直卡在 `[~]`？可能是 implementer 运行超长或 verifier 在 FAIL 循环 |

## 7. 查看结果

### 7.1 运行历史

运行历史页（`/runs`）展示每一轮 implement/verify 的记录：

- **任务描述**：来自 tasks.md
- **阶段**：implement（实现）或 verify（验证）
- **结果**：PASS 或 FAIL
- **耗时**：从开始到完成的时长
- **Agent**：谁执行的

支持按 agent 筛选。

### 7.2 PASS 率

Dashboard 首页和控制页都展示近 7 天的 PASS 率：

- 只统计 verify 阶段的 PASS/FAIL
- 示例：`PASS 率: 85.7% (12/14)`

### 7.3 任务报告

点击任务卡片可查看**任务报告**——通过 git log 回溯该 task_id 的所有 commit：

- 每轮 IMP 对应一个 commit
- 显示实现思路、变更概要、验证结果、已知局限
- 返工（reopen）任务的反馈也在这里

### 7.4 Loop 实时日志

控制页的 **Loop 日志** 区域展示 Claude 最近的输出——从 session JSONL 读取最近的 assistant 消息，转成 HTML 渲染。可以了解 Claude 正在做什么。

## 8. 合入任务分支

### 8.1 分支列表

分支页（`/branches`）列出所有 `agent/*` 分支：

- **分支名**：格式 `agent/<whoami>/<task_id>-<slug>`
- **合入状态**：已合入 / 未合入
- 支持按 agent 筛选

### 8.2 合入流程

在 Claude Code 中使用 `/task-merge` 或直接说"合入 xxx 分支"：

1. **发现分支**：自动列出未合入的 agent 分支
2. **显示概要**：展示分支上的 commit 和改动文件
3. **验证门禁**：在 agent worktree 中预合入，按 TEST.md 跑手动测试
4. **执行合入**：merge 到 master（本地，不自动 push）
5. **验证结果**：显示合入后状态

### 8.3 验证门禁

合入前必须通过验证：

- **TEST.md 存在**：按文档执行相关模块的测试步骤
- **TEST.md 不存在**：跳过，直接合入
- **测试不通过**：合入取消，反馈写入 tasks.md，任务标记 `[r]`

> 人在合入前只需做最终审查——verifier 已经在 Loop 里验证过编译和自动测试了。

## 9. 项目设置

### 9.1 编辑配置

设置页（`/settings`）可以编辑：

| 配置项 | 说明 |
|--------|------|
| 项目名称 | 用于 Dashboard 显示和项目识别 |
| Agent 名称 | 谁的任务谁做 |
| MCP 端口 | 主工程端口（默认 8080）、Agent 端口（默认 9080）|
| 项目类型 | Generic / Unity+ToLua / Python Server |
| 数据仓库路径 | 配表/数据工程的 git 路径（可选）|

修改端口后会自动重新生成 MCP 配置并同步到 agent worktree。

### 9.2 项目类型预设

切换项目类型会自动调整验证文档模板。三种预设：

| 预设 | 适用场景 |
|------|----------|
| **Generic** | 通用项目，Bash 命令 + CLI 验证 |
| **Unity+ToLua** | Unity 游戏项目，MCP 编译 + Lua 热加载 + runtime-test |
| **Python Server** | Python Web/服务端，pytest + curl + WebFetch |

### 9.3 验证文档在线编辑

Dashboard 提供 verifier skills 和 TEST.md 的在线编辑功能。修改后保存即生效，loop-verify 下一次运行时使用最新版本。

### 9.4 模板重新部署

如果 templates/ 中的模板有更新（如新增了 skill），点击 **重新部署** 将所有模板文件同步到 `.claude/` 目录。**注意**：TEST.md 会被覆盖，如有自定义内容请先备份。

## 10. 移除项目

如果不希望再对某个项目使用 Loop Engineering，可以在设置页执行 **Teardown**：

**会删除的内容**：
- Agent git worktree 目录
- 数据仓库 worktree（如果有）
- `loop-config.yaml`
- 注册表中的项目条目

**不会影响的内容**：
- 主项目文件（`.claude/`、`.mcp.json`、源代码等）全部保留
- git 仓库不受任何影响

> 操作需输入项目名确认，防止误删。

## 11. 多项目管理

### 11.1 项目切换

Dashboard 顶部有项目下拉菜单，列出所有已注册的项目：

```
[ PVPProject8  ▾ ]    ← 点击切换
  PVPProject8
  loop-engineering
```

切换后所有页面（任务、控制、分支等）都切换到对应项目。

### 11.2 独立启停

每个项目的 Loop **完全独立**：

- 可以同时运行多个项目的 Loop
- 每个项目有自己的终端窗口、心跳、控制信号
- 暂停项目 A 不影响项目 B
