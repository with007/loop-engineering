# 手动测试指南 — loop-engineering

> 项目类型: Python Server | 由 loop-test-init 定制
>
> 本文件是**测试方法论参考**，不是必须全部执行的清单。
> 说明项目中可用的测试手段，并提供各模块的测试示例。
> 合入时根据变更涉及哪些模块，选择相关手段来测试。
> 重新运行 `loop setup` 会覆盖 — 有自定义内容请备份。

## 测试工具

| 工具 | 用途 |
|------|------|
| 浏览器 | 打开页面、点击导航、走交互流程、视觉检查 |
| 终端 | pip install、CLI 命令执行、进程管理（Ctrl+C 停止） |
| 终端 (Desktop) | 编译 Rust 桌面托盘程序、运行自动化测试、查看日志 |

## 测试手段

以下是本项目中可用的测试方法。合入时根据变更模块选择相关手段。

### 环境准备

```bash
pip install -e ".[ui]"
```

项目依赖：`pyyaml`、`jinja2`（核心）；`fastapi`、`uvicorn`、`python-multipart`（可选，仪表盘需要）。

### 启动服务

| 入口 | 命令 | 说明 |
|------|------|------|
| Web 仪表盘 | `loop ui start --port 8765` | 启动后浏览器打开 http://127.0.0.1:8765 |
| CLI 工具 | `loop --help` | 验证 CLI 子命令可用 |

### API 端点验证

API 通过页面间接验证：页面数据正常加载即说明对应 API 正常。不需要单独检查 HTTP 状态码。

常用端点及其对应页面：
- `/api/projects/list` → 概览页项目统计
- `/api/tasks/list` → 任务列表

### Web 页面验证

浏览器打开页面，确认：
- 无白屏、无 500 错误
- 关键元素正常渲染（导航、表格、图表等）
- 核心交互流程可走通

关键页面：
- `/` — 概览，展示项目统计（通过率图表、近期运行记录）。**操作**: 点击导航栏"概览"，确认图表渲染、数据加载正常
- `/tasks` — 任务列表，显示 tasks.md 中的任务状态和分配情况。**操作**: 点击导航栏"任务"，确认任务卡片正常显示、状态标签正确
- `/runs` — 运行历史，展示所有 IMP/VFY 运行记录。**操作**: 点击导航栏"历史"，确认列表加载、筛选功能正常
- `/control` — 循环控制，管理 agent 循环的启停和心跳状态。**操作**: 点击导航栏"控制"，确认控制按钮可点击、状态信息正确
- `/settings` — 配置编辑，在线编辑 VERIFY.md 和 TEST.md。**操作**: 点击导航栏"设置"，确认编辑器加载、保存功能正常

### CLI 验证

在终端运行 `--help` 确认子命令正常显示，执行核心子命令验证输出正确、无乱码。

## 测试完成后

1. 停止服务（Ctrl+C 或 kill 进程）
2. 切回主 worktree 重新安装依赖：
   ```bash
   pip install -e ".[ui]"
   ```

## 模块测试示例

以下是各模块的测试示例。合入时**只执行变更涉及模块**的测试，未涉及的跳过。

### 仪表盘页面

**适用场景**: 变更涉及仪表盘 HTML 模板、前端 JS、页面路由时执行。

**步骤**:
1. 运行 `loop ui start --port 8765`（自动打开浏览器）
2. 点击导航栏各标签（概览、任务、历史、控制、设置），确认每个页面正常加载
3. 停止服务

**预期**: 所有页面正常加载，无白屏或 500 错误

### CLI 命令

**适用场景**: 变更涉及 CLI 参数解析、子命令、配置读写时执行。

**步骤**:
1. 运行 `loop --help`
2. 运行 `loop config show`
3. 确认输出格式正确、无乱码

**预期**: CLI 命令正常执行，输出格式正确

### API 端点

**适用场景**: 变更涉及 API 路由、请求处理、数据模型时执行。

**步骤**:
1. 运行 `loop ui start --port 8765`（自动打开浏览器）
2. 确认概览页项目统计数据正常显示（验证 `/api/projects/list`）
3. 点击导航栏"任务"，确认任务列表正常显示（验证 `/api/tasks/list`）
4. 停止服务

**预期**: 各页面数据正常加载，无空白或加载失败

### Desktop 应用（托盘程序）

**适用场景**: 变更涉及 `desktop/src/` 下的 Rust 代码时执行。

**步骤**:
1. 进入 `desktop/` 目录，运行 `cargo build --release`，等待编译完成，确认无 error
2. 运行自动化测试：`cargo run --release --bin test-layer1`，确认输出 `ALL CHECKS PASSED`
3. （可选）直接启动程序验证设置窗口：运行 `target/release/loop-dashboard.exe --test`，然后查看 `target/release/dashboard.log`，确认末尾有 `ALL TESTS PASSED` 且中间有 `GlutinWindowContext: ready`、`WindowState: CJK font loaded`
4. 停止程序：`loop-dashboard.exe` 是托盘程序，Ctrl+C 无效。在任务管理器中结束进程，或右键系统托盘图标选"退出"

**预期**: 编译无错误，自动化测试全部通过。设置窗口 GL 渲染正常（日志中 GL context 创建成功），CJK 字体加载成功

### Layer 2 人工验证

**适用场景**: 托盘菜单结构、设置窗口交互等需要人工判断的变更。

**步骤**:
1. 启动 `target/release/loop-dashboard.exe`（无 `--test` 参数，正常模式）
2. 右键系统托盘图标 → 弹出菜单
3. 点击"设置..." → 确认窗口显示（深灰色背景、中文标题"Loop Dashboard 设置"）
4. 修改端口号、切换开机自启 → 点击"保存" → 确认不闪烁、值不跳回
5. 点击 X 关闭设置窗口 → 确认程序不退出（托盘图标仍在）
6. 点击"退出" → 确认进程完全终止
7. 如果配置了项目：检查每个项目子菜单出现，Dashboard 和 Loop 控制按钮根据运行状态正确显示/隐藏

**预期**: 菜单即时响应、设置窗口正常渲染无白屏、编辑不掉字、关闭窗口不退出程序

> 如果变更涉及以上未列出的新模块，参照"测试手段"中的方法推导合适的测试步骤。
> 测试通过后建议将新模块的测试示例添加到本文档。
