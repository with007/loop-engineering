# loop-test-init

初始化/完善项目的 VERIFY.md 和 TEST.md，从通用模板产物定制为项目专属文档。

## 触发

```
/loop-test-init
```

## 流程

### 1. 深入理解项目

先花时间理解项目本身，不要上来就改文档：

- 读项目入口点（`setup.py` / `pyproject.toml` / `package.json` / `go.mod` 等）
- 理解项目的技术栈、运行方式、部署方式
- 了解项目是否有测试框架、测试文件、CI 配置
- 如果项目有 Dockerfile、数据库迁移脚本、多语言构建等特殊流程，记下来

### 2. 读当前文档

读项目根目录的 VERIFY.md 和 TEST.md。

区分两种内容：
- **模板指导性描述**：自然语言段落说明"这步做什么、常见选项是什么"——尚未定制
- **用户已填值**：具体的命令、参数、场景——与模板默认值不同，保留

判断依据：如果某条命令和模板默认值完全一样（如 `python -m pytest || echo skipped`），视为未定制。

### 3. 定制 VERIFY.md

基于项目实际情况：

- 把指导性描述替换为项目实际命令和参数
- 如果步骤不适用（如 Python 项目无模板文件），删除 "模板检查" 步骤
- 如果项目有模板未覆盖的流程（如 Docker 构建、数据库迁移），新增步骤
- 如果项目无测试，在测试步骤标注 "项目暂无测试" 而非空着
- 代码逻辑检查步骤保留（所有项目通用）

### 4. 定制 TEST.md

基于项目实际情况：

- 环境准备：填写项目实际的安装命令（如 `pip install -e ".[ui]"`）
- 启动项目：填写实际的启动命令和服务地址
- 运行时验证：根据项目类型（Web/CLI/纯库/Unity）填写具体验证流程
- 测试完成后：填写 worktree 环境恢复命令
- 如果项目有预设测试场景，如实填写；没有则标注通用建议

### 5. 写回

将定制完成的 VERIFY.md 和 TEST.md 写回项目根目录。

规则：
- 用户已修改的值不覆盖
- 指导性描述被替换为具体内容
- 文档顶部保留覆盖警告（`loop setup` 会覆盖）
- 新增步骤保持文档可读性——不是只写命令，要写清楚为什么

## 禁止

- 不要修改框架级通用验证逻辑（task-runner 处理）
- 不要删除 "代码逻辑检查" 步骤
- 不要编造不存在的测试框架或命令
