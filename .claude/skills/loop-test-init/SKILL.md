# loop-test-init

初始化/定制/校验项目的测试文档（VERIFY.md / TEST.md），或新增项目类型模板。

## 三种模式

```
/loop-test-init               → 模式 1（默认）：定制当前项目
/loop-test-init --check        → 模式 2：校验当前项目文档是否符合规范
/loop-test-init --new-type     → 模式 3：新增一个项目类型的模板
```

---

## 模板规范（9 条）

所有操作以此 9 条为基准。

### 1. 方法论声明

TEST.md 是**测试方法论参考**，不是必须全部执行的清单。头部必须声明这一点。合入时只测变更涉及模块，未涉及的跳过。

### 2. 给人的文档不要自动化口吻

TEST.md 的读者是人。**禁止出现**：MCP、refresh_unity、read_console、runtime-test skill、curl 代码块、WebFetch、DevTools Network 面板。这些是 verifier/agent 的内部知识。

### 3. 间接验证优于直接暴露技术细节

API 验证不要求人看 HTTP 状态码和 JSON——页面数据正常即 API 正常。用"打开页面确认数据加载"替代 `curl` 命令。

### 4. 占位符驱动

模板只依赖 `{{ project_name }}` 一个变量，其余用 `<填写xxx>` 占位符。`loop-test-init` 负责识别并替换所有占位符。定制完成后**不允许残留任何 `<填写...>` 占位符**。

### 5. 定制提示集中到开头

定制说明只在头部出现一次（"`loop-test-init` 会根据项目实际情况定制所有内容"）。各节内部**禁止**散落"可增删"提示。模块示例区的"项目初始化时 loop-test-init 会根据项目实际模块填充以下示例"是唯一例外——它说明该区域的来源。

### 6. 头部格式统一

所有模板头部格式：
```
# <文档类型> — {{ project_name }}

> 项目类型: <TypeName>
>
> 本文件是 <TypeName> 类型的默认模板。`loop-test-init` 会根据项目
> 实际情况定制所有内容：增删章节、调整验证手段、替换占位符。
>
> 本文件<角色说明>...
> <附加说明行>
> 重新运行 `loop setup` 会覆盖 — 有自定义内容请备份。
```

- TEST.md.j2 角色说明：`本文件是**测试方法论参考**，不是必须全部执行的清单。`
- VERIFY.md.j2 角色说明：`本文件定义项目特定的自动化验证步骤。verifier 子代理按此执行。`

### 7. VERIFY.md 变更驱动

VERIFY.md 不是固定步骤表。头部声明"验证范围由变更驱动 — 只执行涉及模块的步骤，未涉及的跳过"。每个模块验证示例标注**触发条件**（如"变更涉及 `.cs` 文件"）。

### 8. 人机分工

- **TEST.md**：给人看，聚焦 verifier 无法判断的内容（画面表现、帧率、UI 交互）。编译检查、自动化测试标注为 verifier 已负责。
- **VERIFY.md**：给 verifier 看。头部必须声明"通用验证（diff 审查、变更范围确认等）由 task-runner 框架统一处理"——verifier 不做框架已做的事。
- TEST.md 不出现自动化工具名；VERIFY.md 不出现"请人手动确认"等手动口吻。

### 9. 文档纳入版本控制

VERIFY.md 和 TEST.md 在项目根目录，受 git 管理。模板 .j2 文件在 `templates/verify/<type>/` 下。

## 模式 1：定制（默认）

### 触发

```
/loop-test-init
```

### 流程

#### 1. 深入理解项目

先花时间理解项目本身，不要上来就改文档：

- 读项目入口点（`setup.py` / `pyproject.toml` / `package.json` / `go.mod` 等）
- 理解项目的技术栈、运行方式、部署方式
- 了解项目是否有测试框架、测试文件、CI 配置
- 了解项目的核心模块：有哪些页面、API、CLI 命令等
- 如果项目有 Dockerfile、数据库迁移脚本、多语言构建等特殊流程，记下来

#### 2. 读当前文档

读项目根目录的 VERIFY.md 和 TEST.md。

文档中的占位符格式：`<填写xxx>`。这些是需要定制的部分。

区分两种内容：
- **占位符**（`<填写xxx>`）→ 需要替换为项目实际值
- **已填充内容**（不是 `<填写...>` 格式）→ 可能已被定制，根据实际情况判断是否更新

#### 3. 定制 VERIFY.md

**3a. 调整测试工具列表**

测试工具表下方有提示说明默认工具及增删规则。根据项目实际情况：
- 纯 CLI 项目：去掉 curl、WebFetch
- 有 Docker 的项目：加上 `docker` / `docker-compose` 命令
- 有数据库的项目：加上数据库客户端命令
- 有额外工具链的：按需添加

**3b. 逐个替换占位符**：

- `<填写测试框架>` → 项目的测试框架名（pytest / unittest / 无则标注"暂无"）
- `<填写测试命令>` → 实际测试命令（如 `python -m pytest`）
- `<填写安装命令>` → 实际安装命令（如 `pip install -e ".[ui]"`）
- `<填写 CLI 入口命令>` → CLI 入口（如 `loop --help`）
- `<填写启动命令>` → 服务启动命令（如 `loop ui start --port 8765 --no-browser`）
- `<填写关键 API URL>` → 关键 API 端点（如 `/api/projects/overview`）
- `<填写页面 URL>` / `<填写关键元素>` → 页面和关键元素名

模块验证示例部分：
- 模板自带的是通用示例占位（`<填写模块名>` / `<填写触发条件>` / `<填写验证步骤>`）
- 根据项目实际模块替换为具体内容
- 如果模板中的示例不适用（如项目无 Web 页面），删除该示例
- 如果项目有模板未覆盖的模块（如 Docker、数据库迁移），新增示例

#### 4. 定制 TEST.md

**4a. 调整测试工具列表**

同 VERIFY.md，根据项目实际情况增删工具。

**4b. 逐个替换占位符**：

- `<填写安装命令>` → 实际安装命令
- `<填写启动命令>` → 服务启动命令
- `<填写服务地址>` → 服务 URL（如 `http://127.0.0.1:8765`）
- `<填写 CLI 入口命令>` → CLI 入口命令
- `<填写 API 路径>` / `<填写对应页面>` / `<填写说明>` → 端点清单
- `<填写各页面的路径、功能说明和操作方式>` → 根据项目实际页面填充：
  - 每页的路径（如 `/tasks`）
  - 功能描述（如"任务列表"）
  - 操作方式（如"点击导航栏'任务'"）
  - 要检查的内容（如"确认任务卡片正常显示"）
- `<填写 CLI 入口命令>` → 实际 CLI 入口

模块测试示例部分：
- 同 VERIFY.md，模板自带的是通用示例占位
- 根据项目实际模块替换，删除不适用的，新增缺失的

**4c. 确保无自动化口吻（规范 2）**

检查 TEST.md 全文，确保：
- 不出现 MCP / refresh_unity / read_console / runtime-test skill 等技术工具名
- 编译检查用"等待编译完成，检查 Console 无 error"，不用"refresh_unity → read_console"
- Lua 测试用"确认 Console 中标记全部 PASS"，不用"调用 runtime-test skill"
- API 验证用"打开页面确认数据加载"，不用 curl 命令

#### 5. 写回

将定制完成的 VERIFY.md 和 TEST.md 写回项目根目录。

规则：
- 所有 `<填写xxx>` 占位符必须被替换（不留未填充的占位符）
- 已填充的内容如果准确则保留，如果过时则更新
- 文档头部保留覆盖警告（`loop setup` 会覆盖）
- 新增步骤保持文档可读性——不是只写命令，要写清楚为什么

#### 6. 自检

定制完成后，**自动按模式 2 的规则逐条自检**。如有未通过的项，当场修复后再输出最终报告。

### 禁止

- 不要修改框架级通用验证逻辑（task-runner 处理）
- 不要删除 VERIFY.md 的"代码逻辑检查"步骤
- 不要删除 VERIFY.md 的"通用验证由 task-runner 框架统一处理"声明
- 不要编造不存在的测试框架或命令
- 不要留未填充的 `<填写xxx>` 占位符
- **不要给 TEST.md 添加自动化工具名**

## 模式 2：校验

### 触发

```
/loop-test-init --check
```

### 流程

读 TEST.md 和 VERIFY.md（先确认文件在项目根目录存在），逐条对照规范输出校验报告。

### 校验规则表

#### TEST.md（检查规范 1-6、8-9）

| # | 规范 | 通过条件 | 失败时说明 |
|---|------|----------|------------|
| 1 | 方法论声明 | 头部含"测试方法论参考" | `缺少方法论声明，应加：本文件是测试方法论参考，不是必须全部执行的清单` |
| 2 | 无自动化口吻 | 不含 MCP / refresh_unity / read_console / runtime-test / curl 代码块 | `第N行发现 'xxx'：这是自动化工具名/TEST.md 是给人看的` |
| 3 | 间接验证 | 如有 API 验证章节，用"页面"描述而非 HTTP 状态码 | `API 验证应改为页面间接验证：打开页面确认数据加载，而非检查状态码` |
| 4 | 无残留占位符 | 不含 `<填写...>` | `第N行残留占位符 '<填写xxx>'：应替换为项目实际值` |
| 5 | 定制提示集中 | 只有头部有"loop-test-init"字样，各节内无"可增删"提示 | `第N行有散落的定制提示：应集中在头部` |
| 6 | 头部格式正确 | 含项目类型行、默认模板声明、角色说明、覆盖警告 | `头部格式不完整：缺少 '重新运行 loop setup 会覆盖' 警告` |
| 8 | 人机分工 | 无自动化工具名（同规范2）；编译/Lua 测试标注为 verifier 已负责 | `编译检查应写'等待编译完成，检查 Console'，而非'通过 MCP refresh_unity'` |
| 9 | 位置正确 | 文件在项目根目录 | `TEST.md 不在项目根目录，应在 <project_root>/TEST.md` |

#### VERIFY.md（检查规范 4-9）

| # | 规范 | 通过条件 | 失败时说明 |
|---|------|----------|------------|
| 4 | 无残留占位符 | 不含 `<填写...>` | `第N行残留占位符 '<填写xxx>'：应替换为项目实际值` |
| 5 | 定制提示集中 | 只有头部有"loop-test-init"字样 | `第N行有散落的定制提示` |
| 6 | 头部格式正确 | 含项目类型行、默认模板声明、角色说明、覆盖警告 | `头部格式不完整` |
| 7 | 变更驱动 | 头部声明"验证范围由变更驱动"；模块示例有"触发条件" | `缺少变更驱动声明 / 模块示例无触发条件` |
| 8 | 人机分工 | 含"通用验证由 task-runner 框架统一处理"声明 | `缺少框架边界声明：通用验证（diff 审查、变更范围确认等）由 task-runner 框架统一处理` |
| 9 | 位置正确 | 文件在项目根目录 | `VERIFY.md 不在项目根目录` |

## 模式 3：新增项目类型模板

### 触发

```
/loop-test-init --new-type
```

### 流程

#### 1. 询问项目特征

向用户了解新类型的核心特征：

- 项目类型名（如 "Rust CLI"）
- 技术栈和运行方式
- 验证手段：有编译？有测试框架？有 Web 服务？有 CLI？
- 特殊工具链（Docker、数据库、移动端等）

#### 2. 生成 TEST.md.j2

按规范生成，结构参照现有模板：

```
# 手动测试指南 — {{ project_name }}

> 项目类型: <TypeName>
>
> 本文件是 <TypeName> 类型的默认模板。`loop-test-init` 会根据项目
> 实际情况定制所有内容：增删章节、调整验证手段、替换占位符。
>
> 本文件是**测试方法论参考**，不是必须全部执行的清单。
> 说明项目中可用的测试手段，并提供各模块的测试示例。
> 合入时根据变更涉及哪些模块，选择相关手段来测试。
> 重新运行 `loop setup` 会覆盖 — 有自定义内容请备份。

## 测试工具
## 测试手段
### 环境准备
### <编译/构建检查>        ← 人的视角，不提自动化工具名
### <启动方式>
### <运行时验证>           ← 聚焦视觉/交互，verifier 无法判断的
## 测试完成后
## 模块测试示例
```

**硬约束**：
- 头部格式与规范 6 完全一致
- 不出现任何自动化工具名（规范 2）
- 所有待定制内容用 `<填写xxx>`（规范 4）
- 无散落的定制提示（规范 5）

#### 3. 生成 VERIFY.md.j2

按规范生成：

```
# 验证指南 — {{ project_name }}

> 项目类型: <TypeName>
>
> 本文件是 <TypeName> 类型的默认模板。`loop-test-init` 会根据项目
> 实际情况定制所有内容：增删章节、调整验证手段、替换占位符。
>
> 本文件定义项目特定的自动化验证步骤。verifier 子代理按此执行。
> 验证范围由变更驱动 — 只执行涉及模块的步骤，未涉及的跳过。
> 重新运行 `loop setup` 会覆盖 — 有自定义内容请备份。
>
> 通用验证（diff 审查、变更范围确认等）由 task-runner 框架统一处理。

## 测试工具
## 自动化验证手段
### 1. <编译/构建>
### 2. <测试运行>
### 3. 代码逻辑检查
### 4. <运行时验证>       ← verifier 视角：curl/WebFetch/CLI 命令
## 模块验证示例
```

**硬约束**：
- 头部含"通用验证由 task-runner 框架统一处理"（规范 8）
- 头部含"验证范围由变更驱动"（规范 7）
- 模块示例有"触发条件"（规范 7）
- 代码逻辑检查步骤不可删除
- 所有待定制内容用 `<填写xxx>`（规范 4）

#### 4. 写入

文件命名：`TEST.md.j2` 和 `VERIFY.md.j2`，放到 `templates/verify/<type-slug>/` 下。`type-slug` 取项目类型名的 kebab-case（如 `rust-cli`）。

#### 5. 自检

新模板写入后，**以新类型身份模拟运行一次模式 2 校验**，确认 9 条规范全部通过。
