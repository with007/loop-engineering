# Loop Engineering — Claude Rules

## 1. Bug 修复流程

如果是报错/bug 而不是新需求，**不要直接开始修改代码**。流程：

1. 先读相关代码，定位问题根因
2. 告诉用户：发现了什么问题、是什么原因导致的
3. 说明打算怎么修复
4. 用户确认后再动手

这避免对 bug 理解偏差导致越修越远。

## 2. 修改后自测（必须用真实环境）

修改完代码之后，**必须用生产环境的真实命令和流程跑通测试**，禁止用简化替代品。测试不通过不能提交。

### 什么是"真实环境"

- **进程/窗口相关**：必须启动真实的 `cmd /k claude --dangerously-skip-permissions`，不能用 `echo hello` 或 `timeout` 代替
- **API 端点**：写脚本调用实际的 API URL，或直接调用底层函数并验证返回值
- **模板**：用 Jinja2 加载真实模板文件渲染，检查所有关键字段存在

### 禁止

- 用 `echo` / `timeout` / `ping` 等替代 `claude` 命令来"模拟"长运行进程
- 用简化参数测试 PowerShell/CMD 行为，必须以实际生产命令为准
- 只测 happy path 不测异常情况（如进程被杀后 PID 文件清理）
